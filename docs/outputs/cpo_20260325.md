
### 방향 1: Full-Page Tiling (EYE-08) ← 지금 바로
- 페이지 전체 높이 측정 후 1600px 단위로 타일링 (100px 오버랩)
- 최대 타일: 10개 (=16,000px), 초과 시 상단/중간/하단 샘플링
- **크레딧 단위 변경**: 1 URL → 1 타일 (1280×1600 분석 1회)
- 짧은 페이지(800px) = 1 크레딧 / 긴 랜딩(4800px) = 3 크레딧

### 방향 2: Multi-Viewport (EYE-09) ← 다음 스프린트
- `viewport` 파라미터: `desktop`(1280) | `tablet`(768) | `mobile`(375) | `all`
- 각 뷰포트 × full-page tiling → CSS 미디어쿼리 자동 적용
- `all` = 3 뷰포트 세트 = 크레딧 3배 (투명 과금)

### 방향 3: Site Crawl (EYE-10) ← 그 다음
- `visual_crawl(url, depth=1)` — 같은 도메인 전체 링크 자동 순회
- 각 페이지 × 3 뷰포트 × full-page tiling
- GitHub Action 연동: PR마다 변경된 페이지만 체크

### 방향 4: GitHub PR Integration (EYE-11) ← Site Crawl 후
- PR 올라올 때마다 변경된 경로만 타겟 QA
- 리포트: "17페이지 중 3페이지에서 6개 버그"

## DOM Audit Engine 위치 정의

First Principles 분석 결과 수용 — 단, 목적을 명확히:

```
DOM Audit  = 빠른 사전 스캔 (overflow/broken image 즉시 탐지, 추가 비용 $0)
Screenshot = 물리적 시각 증거 (사용자가 눈으로 확인할 수 있는 캡처)
AI 분석    = DOM Audit이 flagged한 타일에 집중 (정확도 향상)
```

**스크린샷을 제거하지 않는다.** CEO 방침: "물리적으로 비주얼을 체크하는 것이 목적".
DOM Audit은 타일 우선순위 결정 + AI 프롬프트 강화에 활용.

```typescript
const pageHeight = await page.evaluate(() => document.body.scrollHeight);

// 2. 타일 계산 (1600px 단위, 100px 오버랩)
const TILE_HEIGHT = 1600;
const OVERLAP = 100;
const tileCount = Math.min(10, Math.ceil(pageHeight / (TILE_HEIGHT - OVERLAP)));

// 3. 타일별 clip 스크린샷
for (let i = 0; i < tileCount; i++) {
  const y = i * (TILE_HEIGHT - OVERLAP);
  const shot = await page.screenshot({
    encoding: "base64", type: "jpeg", quality: 80,
    clip: { x: 0, y, width: 1280, height: Math.min(TILE_HEIGHT, pageHeight - y) }
  });
  // 4. 각 타일 Gemini 분석 (순차 or 최대 2개 병렬)
}

// 5. 결과 머지: has_issues = tiles.some(t => t.has_issues)
```
