## 사업성 (Business Viability) 분석

### 핵심 질문: 신뢰 없이 B2A2H는 작동하지 않는다

현재 비즈니스 모델의 전제는 **"에이전트가 신뢰해서 추천 → 사람이 결제"**다.
그런데 지금 구조에서 에이전트는 같은 URL을 3번 호출하면 `false / error / true`를 받는다.
에이전트는 불확실한 도구를 추천하지 않는다. B2A2H 전환 퍼널의 첫 단계가 막힌다.

```
지금 퍼널:
에이전트 사용 → 결과 불신 → 재호출 → 또 다른 결과 → 도구 포기 → 결제 없음

목표 퍼널:
에이전트 사용 → 수치 근거 결과 → 신뢰 → 반복 사용 → 추천 → 결제
```

### 수익 지속성

| 시나리오 | 순수 Screenshot AI | DOM + Screenshot 하이브리드 |
|---------|-------------------|--------------------------|
| 초기 인상 | "Cool, AI가 화면을 봐준다" | 동일 (스크린샷 유지) |
| 2주 후 | False positive 누적 → 이탈 | 수치 근거 결과 → 잔존 |
| 1개월 후 | 신뢰 붕괴 → 유료 전환 불가 | 반복 사용 → 유료 전환 |
| Gemini 모델 업그레이드 시 | 경쟁력 소멸 가능 | DOM 레이어는 AI 독립적 |

### 경쟁 방어력 (Defensibility)

- **Applitools·Percy·Chromatic**: 스크린샷 diff 전문. MCP 없음. 에이전트 네이티브 없음.
- **Lighthouse·axe-core**: DOM 검사. 에이전트가 실행 불가. MCP 없음.
- **Playwright MCP·Stagehand**: "손" (조작) 전문. 눈(QA) 없음.
- **Gemini·Claude Vision 네이티브**: 스크린샷 분석은 흡수 가능. DOM + MCP 파이프라인은 흡수 불가.

> **DOM + MCP의 조합은 대형 플레이어가 빠르게 복제할 수 없다.**
> DOM 정확도는 엔지니어링이고, MCP 생태계 포지셔닝은 타이밍이다.

### 요금제와 Unit Economics

| 방식 | 타일당 AI 비용 | 타일당 DOM 비용 | 합계 | Pro 마진 |
|------|--------------|----------------|------|---------|
| 현재 (AI only) | $0.000012 | $0 | $0.000012 | 99.4% |

DOM이 먼저 필터링하면 AI를 호출할 타일 수가 줄어 **원가가 오히려 낮아진다.**

## 현실성 (Feasibility) 분석

### 기술 구현 난이도

| 항목 | 난이도 | 근거 |
|------|--------|------|
| JS 인젝션 (CF Browser) | 낮음 | `page.evaluate()` — 이미 사용 중 |
| DOM overflow 감지 | 낮음 | `scrollWidth > clientWidth` 등 웹 표준 API |
| Broken image 감지 | 낮음 | `img.complete && img.naturalWidth === 0` |
| viewport 파라미터 | 낮음 | `page.setViewport({ width: 375 })` 한 줄 |
| AI 번역 통합 | 낮음 | 이미 있는 Gemini 호출에 DOM 결과 텍스트 추가 |
| 기존 코드 유지 | 낮음 | 스크린샷 파이프라인 제거 없이 DOM 레이어 추가 |

**예상 구현 기간: 1일 (현재 인프라 100% 재활용)**

### 리스크 목록

| 리스크 | 확률 | 영향 | 대응 |
|--------|------|------|------|
| CF Browser JS 실행 제한 | 낮음 | 높음 | 이미 `page.evaluate` 사용 중 — 이미 검증됨 |
| DOM 측정 false positive | 낮음 | 중간 | 임계값 설정 (1px 오차 허용) |
| DOM으로 못 잡는 버그 존재 | 확실 | 낮음 | 스크린샷 AI가 보완 — 구조에 이미 포함 |
| 기존 tiling 코드 충돌 | 없음 | 없음 | DOM은 스크린샷 전 단계에 삽입 |

### 롤백 가능성

DOM 레이어는 기존 파이프라인에 **추가**하는 구조다. 제거하려면 코드 3~5줄 삭제.
롤백 비용 = 0. 위험 없음.

## 다음 에이전트 인수인계

**Dev 에이전트 → EYE-08 구현 지시:**

```typescript
// 1. page.evaluate로 scrollHeight 획득
const pageHeight = await page.evaluate(() => document.body.scrollHeight);

// 2. 타일 계산 (1600px 단위, 100px 오버랩)
const TILE_HEIGHT = 1600;
const OVERLAP = 100;
const tileCount = Math.min(10, Math.ceil(pageHeight / (TILE_HEIGHT - OVERLAP)));

// 3. 타일별 clip 스크린샷
for (let i = 0; i < tileCount; i++) {
  const y = i * (TILE_HEIGHT - OVERLAP);
  const shot = await page.screenshot({
    encoding: "base64", type: "jpeg", quality: 80,
    clip: { x: 0, y, width: 1280, height: Math.min(TILE_HEIGHT, pageHeight - y) }
  });
  // 4. 각 타일 Gemini 분석 (순차 or 최대 2개 병렬)
}

// 5. 결과 머지: has_issues = tiles.some(t => t.has_issues)
// 6. 크레딧 차감: tileCount만큼
```

**응답 구조 변경:**
- `tiles_analyzed: number` 필드 추가
- `credits_used: number` 필드 추가 (= tiles_analyzed)
- `issues[]`에 `tile_index` 필드 추가 (어느 타일에서 발견했는지)
